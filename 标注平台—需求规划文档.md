
## 一、需求概要

当前「监控自动分析 Agent」已完成 告警触发 → 故障分析 → 企微通知 的全链路打通，但存在三个核心痛点：

1. **分析质量无量化评估** — 缺乏系统性质量度量，无法追踪准确度变化趋势
2. **缺乏反馈闭环** — 错误/遗漏无法有效反馈给 Agent，同类问题反复出现
3. **优化方向不明确** — 缺少结构化标注数据支撑 skill 优化和 prompt 调优

本平台围绕 Case 的完整生命周期构建：**录入 → 标注 → 数据集维护**，将 Agent 的"黑盒输出"变为"可度量、可追踪、可优化"的闭环流程。

## 三、数据模型

### 3.1 Case 表（原始录入数据）

| 字段                | 类型          | 说明                                   | 示例                         |
| ----------------- | ----------- | ------------------------------------ | -------------------------- |
| `case_id`         | STRING (PK) | 唯一标识，自动生成                            | `c_20260617_001`           |
| `run_time_ms`     | INTEGER     | 运行时间(ms)，用于评判性能优化                    | `194700`                   |
| `tokens_consumed` | INTEGER     | Tokens 消耗，用于评判 token 节省优化            | `693201`                   |
| `source`          | TEXT        | Case 来源（CommitId/监控告警链接等）            | `https://knot.woa.com/...` |
| `description`     | TEXT        | Case 描述（AI 总结），用于 Case 分类            | 本次告警是弹幕过滤 LLM 子服务...       |
| `category`        | STRING      | Case 分类（Agent 自动分类），评估 Agent 对某类 Case 的覆盖度 | `Commentary`               |
| `agent_input`     | TEXT        | Agent 输入 (query)                     | app_id：2107... 告警内容...     |
| `agent_output`    | TEXT        | Agent 输出 (answer)                    | 一句话结论... 建议... 因果链...      |
| `status`          | STRING      | 状态：`pending` / `annotated`           | `pending`                  |
| `created_at`      | DATETIME    | 创建时间                                 | `2026-06-17 17:11:00`      |
| `updated_at`      | DATETIME    | 更新时间（Case 状态变更时自动更新，如标注完成后从 pending 变为 annotated） | `2026-06-17 18:30:00`      |

### 3.2 Annotation 表（标注数据）

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `annotation_id` | STRING (PK) | 唯一标识，自动生成 | `a_20260617_001` |
| `case_id` | STRING (FK) | 关联的 Case ID | `c_20260617_001` |
| `root_cause` | TEXT | 问题根因（人工分析给出的根本原因） | 偶发调用失败... |
| `accuracy_score` | INTEGER | 准确度（0=不合格 / 1=合格 / 2=优秀） | `2` |
| `operability_score` | INTEGER | 可操作性（0=不合格 / 1=合格 / 2=优秀） | `2` |
| `readability_score` | INTEGER | 可读性（0=不合格 / 1=合格 / 2=优秀） | `1` |
| `overall_score` | FLOAT | 整体分数（三项平均，保留两位小数），自动计算 | `1.67` |
| `remark` | TEXT | 备注（描述准确度/方向/输出形式/优化建议等，写预期结果指导 Agent 优化） | 结论准确... |
| `created_at` | DATETIME | 标注时间 | `2026-06-17 18:30:00` |

## 四、API 设计

### 4.1 Case 录入

**`POST /api/cases`**

请求体：
```json
{
  "run_time_ms": 194700,
  "tokens_consumed": 693201,
  "source": "https://knot.woa.com/agent/detail/...",
  "description": "本次告警是弹幕过滤 LLM 子服务...",
  "category": "Commentary",
  "agent_input": "app_id：2107\napp_mark：...",
  "agent_output": "## 一句话结论\n本次告警是..."
}
```

成功响应 `200`：
```json
{
  "case_id": "c_20260617_001",
  "status": "pending",
  "created_at": "2026-06-17T17:11:00"
}
```

> **标注链接规则**：`/annotate/<case_id>`，由前端根据返回的 `case_id` 动态拼接。

缺失必填字段响应 `400`：
```json
{
  "error": "Missing required fields",
  "missing_fields": ["agent_input", "agent_output"]
}
```

### 4.2 Case 列表查询

**`GET /api/cases`**

查询参数：
| 参数 | 说明 |
|------|------|
| `status` | 按标注状态筛选：`pending` / `annotated` |
| `category` | 按 Case 分类筛选 |
| `start_date` / `end_date` | 按创建时间范围筛选（ISO 8601） |
| `sort_by` | 排序字段：`created_at`（默认）/ `category` / `status` |
| `sort_order` | 排序方向：`desc`（默认，最新在前）/ `asc` |
| `page` | 页码（从 1 开始） |
| `page_size` | 每页数量（默认 20，最大 100） |

成功响应 `200`：
```json
{
  "total": 150,
  "page": 1,
  "page_size": 20,
  "cases": [
    {
      "case_id": "c_20260617_001",
      "source": "https://knot.woa.com/...",
      "category": "Commentary",
      "description": "本次告警是弹幕过滤...",
      "status": "pending",
      "created_at": "2026-06-17T17:11:00"
    }
  ]
}
```

### 4.3 标注页面数据

**`GET /api/cases/<case_id>/annotation`**

获取 Case 详情 + 已有标注（若有）。

成功响应 `200`：
```json
{
  "case": {
    "case_id": "c_20260617_001",
    "source": "https://knot.woa.com/...",
    "category": "Commentary",
    "description": "本次告警是弹幕过滤...",
    "run_time_ms": 194700,
    "tokens_consumed": 693201,
    "agent_input": "app_id：2107...",
    "agent_output": "## 一句话结论\n..."
  },
  "annotation": {
    "root_cause": "偶发调用失败",
    "accuracy_score": 2,
    "operability_score": 2,
    "readability_score": 1,
    "overall_score": 1.67,
    "remark": "结论准确..."
  }
}
```

Case 不存在 `404`；Case 存在但无标注时 `annotation` 字段为 `null`。

### 4.4 标注提交

**`POST /api/annotations`**

请求体：
```json
{
  "case_id": "c_20260617_001",
  "root_cause": "偶发调用失败",
  "accuracy_score": 2,
  "operability_score": 2,
  "readability_score": 1,
  "remark": "结论准确，给出了子服务IP。"
}
```

必填字段：`case_id`、`root_cause`、`accuracy_score`、`operability_score`、`readability_score`

**校验规则**：
- `case_id` 必须在 Case 表中存在，否则返回 `404`：`{"error": "Case not found"}`
- `overall_score` 若客户端传入则忽略，服务端自动计算

成功响应 `200`：
```json
{
  "annotation_id": "a_20260617_001",
  "overall_score": 1.67
}
```

同一 Case 重复标注时覆盖更新，返回相同的 `annotation_id`。

缺失必填字段响应 `400`：
```json
{
  "error": "Missing required fields",
  "missing_fields": ["root_cause"]
}
```

> **`overall_score` 处理**：客户端提交时若包含 `overall_score` 字段，服务端将忽略该值并自动计算（三项平均，保留两位小数）。

### 4.5 标注数据查询

**`GET /api/annotations`**

查询参数：
| 参数 | 说明 |
|------|------|
| `start_date` / `end_date` | 按标注时间范围筛选 |
| `category` | 按 Case 分类筛选 |
| `case_status` | 按 Case 的标注状态筛选（`pending` / `annotated`） |
| `keyword` | 按 Case 的 description / agent_output 内容关键词搜索 |
| `accuracy_min` / `accuracy_max` | 准确度分数区间 |
| `operability_min` / `operability_max` | 可操作性分数区间 |
| `readability_min` / `readability_max` | 可读性分数区间 |
| `overall_min` / `overall_max` | 整体分数区间 |
| `page` / `page_size` | 分页 |

成功响应 `200`：返回标注列表（含关联的 Case 概要信息）

### 4.6 数据导出

**`GET /api/export`**

查询参数：
| 参数 | 说明 |
|------|------|
| `format` | `csv` 或 `json` |
| 其他筛选参数 | 同 `GET /api/annotations` |

CSV 格式：一行一条记录，包含 Case 信息 + 标注信息（与 case 数据示例的合并视图一致），使用 **UTF-8 BOM** 编码（确保 Excel 正确显示中文）

JSON 格式：完整数据结构，适合程序化处理

### 4.7 数据统计

**`GET /api/stats`**

查询参数：
| 参数 | 说明 |
|------|------|
| `start_date` / `end_date` | 按时间范围筛选统计 |
| `category` | 按分类筛选 |

成功响应 `200`：
```json
{
  "overview": {
    "total_cases": 150,
    "annotated_cases": 120,
    "pending_cases": 30,
    "annotation_rate": 0.8
  },
  "averages": {
    "accuracy": 1.5,
    "operability": 1.3,
    "readability": 1.6,
    "overall": 1.47
  },
  "category_distribution": [
    {"category": "Commentary", "annotated": 50, "pending": 10, "avg_overall": 1.6},
    {"category": "Pubg", "annotated": 30, "pending": 5, "avg_overall": 1.3}
  ],
  "low_score_cases": [
    {"case_id": "c_001", "category": "Commentary", "overall_score": 0.67, "root_cause": "..."}
  ]
}
```

### 4.8 Case 删除

**`DELETE /api/cases/<case_id>`**

删除指定 Case 及其关联的 Annotation 数据。

成功响应 `200`：
```json
{
  "message": "Case and associated annotations deleted",
  "case_id": "c_20260617_001"
}
```

Case 不存在响应 `404`：
```json
{
  "error": "Case not found"
}
```

### 4.9 Annotation 删除

**`DELETE /api/annotations/<annotation_id>`**

删除指定标注数据，并将关联 Case 状态回退为 `pending`。

成功响应 `200`：
```json
{
  "message": "Annotation deleted, case status reverted to pending",
  "annotation_id": "a_20260617_001",
  "case_id": "c_20260617_001",
  "case_status": "pending"
}
```

Annotation 不存在响应 `404`：
```json
{
  "error": "Annotation not found"
}
```

## 五、前端页面

### 5.1 Case 列表页 (`/cases`)

- 展示 Case 概要信息表格：来源、分类、描述摘要、状态、创建时间
- 提供筛选控件：标注状态、Case 分类、时间范围
- 分页控件（每页 20 条）

### 5.2 标注页面 (`/annotate/<case_id>`)
**Case 信息展示区**（只读）：
- 来源、分类、描述
- 运行时间、Tokens 消耗
- Agent 输入 (query)
- Agent 输出 (answer)

**标注内容填写区**（6 个维度）：

| 维度 | 类型 | 选项/格式 |
|------|------|-----------|
| 问题根因 | 文本输入 | 自由文本 |
| 准确度 | 单选打分 | 0=不合格 / 1=合格 / 2=优秀 |
| 可操作性 | 单选打分 | 0=不合格 / 1=合格 / 2=优秀 |
| 可读性 | 单选打分 | 0=不合格 / 1=合格 / 2=优秀 |
| 整体分数 | 自动计算 | 三项平均，保留两位小数，选择后实时更新 |
| 备注 | 文本输入 | 自由文本 |

**已有标注回显**：页面加载时若 Case 已有标注，则自动回显所有标注数据供修改。

### 5.3 数据统计看板 (`/dashboard`)

- 统计卡片：总 Case 数、已标注数、未标注数、标注完成率
- 平均分数卡片：平均准确度、平均可操作性、平均可读性、平均整体分数
- 分类标注分布图：不同分类的标注数量与平均分
- 低分 Case 列表：整体分数低于阈值（默认 1.0，可通过查询参数 `low_score_threshold` 配置）的 Case，点击跳转至标注页面 `/annotate/<case_id>`
- 时间范围选择器：按选定时间范围重新计算统计指标

## 六、数据用途（反馈闭环）

标注数据沉淀后，用于以下优化方向：

| 用途 | 说明 |
|------|------|
| **Prompt 调优** | 低分 Case 分析 → 发现 Agent 输出中的薄弱环节 → 优化 Prompt |
| **Skill 优化** | 高频错误模式 → 补充知识库 / 调整推理逻辑 |
| **模型微调** | 标注数据集可作为 SFT/RLHF 的训练数据 |
| **质量追踪** | 长期追踪分析准确度变化趋势，评估优化效果 |

## 七、验收标准

| # | 验收标准 | 对应阶段 |
|---|---------|---------|
| 1 | Agent 可通过 `POST /api/cases` 接口提交 Case 信息（含所有字段） | 数据录入 |
| 2 | 接收 Case 后返回 `case_id`，标注链接由前端按规则 `/annotate/<case_id>` 动态拼接 | 数据录入 |
| 3 | 标注页面包含 6 个标注维度（问题根因、准确度、可操作性、可读性、整体分数、备注） | 人工标注 |
| 4 | 准确度/可操作性/可读性支持 0/1/2 三级打分 | 人工标注 |
| 5 | 整体分数自动计算为三项平均分（保留两位小数） | 人工标注 |
| 6 | 标注数据可正常保存、查询（含关键词搜索）和导出（CSV/JSON） | 数据集维护 |
| 7 | 标注链接可在企微内正常访问 | 人工标注 |

## 八、非功能需求

| 需求     | 说明                                          |
| ------ | ------------------------------------------- |
| 技术栈    | Flask (Python 3.11) + SQLite+ 纯 HTML/CSS/JS |
| 部署方式   | Docker 容器化                                  |
| 数据持久化  | 标注数据不丢失，支持定期备份                              |
| 扩展性    | 后续可接入更多 Agent，支持更多 Case 类型                  |
| 认证鉴权   | 标注页面为开放链接访问（无需登录），后续可接入企微 OAuth             |
| API 风格 | RESTful JSON                                |
| API 版本控制 | URI 前缀版本控制，如 `/api/v1/cases`，为后续接口变更留空间    |
| 限流策略   | API 限流（如每 IP 每分钟 60 次请求），防止恶意刷接口             |
| 日志策略   | 记录操作日志（标注提交、Case 删除等）和错误日志，便于审计和排查          |
| 数据库迁移  | 使用 Flask-Migrate / Alembic 管理 schema 变更，避免手动 DDL |
| SQLite 并发 | SQLite 写入并发采用 WAL 模式 + 合理连接池配置，避免 `database is locked` 错误 |
| 企微浏览器兼容 | 标注页面需兼容企微内置浏览器：viewport 适配、CSS 自动前缀、避免使用企微不支持的 JS API |
| Base URL 配置 | 通过环境变量 `BASE_URL` 配置部署域名，用于生成完整的标注链接（如 `https://your-domain.com/annotate/<case_id>`） |
